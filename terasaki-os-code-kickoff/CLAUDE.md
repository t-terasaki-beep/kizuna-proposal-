# terasaki-os-code — 寺嵜流 提案自動生成OS

> このファイルは新リポジトリ `t-terasaki-beep/terasaki-os-code` のルートに置く「OSの頭脳」です。
> Claude Code はセッション開始時にこれを読み、寺嵜流のまま考え・作業します。

## ミッション

株式会社シスコムネット 寺嵜の代わりに、**省エネ・脱炭素の営業提案書を寺嵜流のまま自動生成・量産**する。
顧客の電気データを入力すれば、商談にそのまま出せる「提案スライド + ナレーション台本 + 音声(MP3)」を生み出す。

### 利益目標についての正直な前提
- お金は「商談 → 契約 → 工事」から生まれる。OSが直接振り込むわけではない。
- OSの役割は**提案の質と本数を上げること**。これが売上の上限を押し上げる唯一の正しいレバー。
- 目安: KIZUNA級（工事規模 約2,000〜2,500万円）の案件で粗利を積み上げ、月次の成約を増やす。
  毎月の成約本数 × 平均粗利 = 月次利益。OSは「本数」と「決定率」を底上げする。
- 毎月: ①新規顧客向け提案を増産 ②既存テンプレと数値根拠を更新 ③決まらなかった理由を学習し改善。

## 寺嵜流（提案の型）

KIZUNA HOME提案で確立された型。すべての提案はこの骨格に従う。

1. **挨拶と要点宣言** — 「要点を絞ってお伝えします」。相手の時間を尊重。
2. **現状の可視化** — 年間電気代・使用量(kWh)・稼働パターン・ピーク時間・入居率/稼働率。
   「満床/フル稼働だといくらに増えるか」まで必ず示す（伸びしろを見せる）。
3. **3段階の柱** = **減らす・作る・貯める**
   - 減らす: 窓の遮熱・断熱コーティング（熱の出入りを抑える）
   - 作る: 太陽光発電（kW）
   - 貯める: 蓄電池（kWh）— 夕方〜夜に使う
   - 必ず「省エネ診断の結果を踏まえた組み合わせ」と位置づける。
4. **2プラン提示（A/B）** — 「どちらを選んでも正解」になるよう設計。
   - Aプラン: 補助金活用を最適化。投資総額・補助金・自己負担・回収年数。
   - Bプラン: 削減を最大化（住居部の窓も追加など）。自己負担・回収年数。
5. **削減効果の幅** — 現状ベースと満床/フル稼働時の年間削減額。回収年数の短縮、20年累計効果。
6. **金額に表れない価値** — 停電時の安心（蓄電池の自立運転、特定回路バックアップ時間）。
   顧客の事業特性に紐づけて語る（例: 認知症対応施設 → 環境変化が健康リスク）。
7. **進め方とスケジュール** — 補助金の受付/締切/交付決定/工事の時系列。
   **注意喚起: 交付決定前の契約・着工は補助対象外**。「申請は寺嵜が代行、負担は最小限」。
8. **まとめ3点 + 連絡先** — ①確実に下がる ②実質負担を抑えられる ③もしもの時も守れる。
   「ご判断はお客様のペースで」「最後まで伴走します」。
   連絡先: 電話 043-261-8033 / メール t-terasaki@syscomnet.co.jp

## 口調・文体（tone of voice）

- 丁寧な敬語、誠実、押しつけない。「伴走」「お客様のペースで」が核。
- 専門用語は噛み砕く。数字は根拠とセットで。
- ナレーションは1スライド1メッセージ、耳で聞いて分かる長さ。
- フォント: 見出し Shippori Mincho、本文 Zen Kaku Gothic New。
- 配色: 深緑(--green-deep #143f35)・緑(--green #1f6f5c)・金(--gold #c19a3e)・和紙色(--paper #f7f3e9)。

## 成果物フォーマット

1. **スライド(index.html)** — 単一HTMLの自己完結デッキ。light/darkテーマ交互。`audio/01.mp3...` を同階層参照。
2. **ナレーション台本** — スライド枚数ぶんの日本語テキスト（上の型の順）。
3. **音声(MP3)** — edge-tts `ja-JP-NanamiNeural`、`audio/NN.mp3`。
   （`pip install edge-tts` → `python engine/mp3gen.py`）

## リポジトリ構成（目標）

```
terasaki-os-code/
  CLAUDE.md                 # このファイル（OSの頭脳）
  README.md
  knowledge/                # 寺嵜流ナレッジベース
    methodology.md          # 減らす・作る・貯める の詳細と数式
    subsidies.md            # 補助金カレンダー・要件・締切
    pricing.md              # 機器単価・係数・回収計算の前提
    tone-and-voice.md       # 話し方・敬語・伴走スタイル
  templates/
    deck.html               # KIZUNAベースのスライドテンプレ
    narration.md            # ナレーション構成テンプレ
  engine/
    diagnose.py             # 入力→診断（減/作/貯、回収年、20年累計）
    build_proposal.py       # データ→index.html 生成
    mp3gen.py               # 台本→MP3 (edge-tts)
  customers/
    <顧客名>/               # 案件ごとの入力・出力・実績
  data/
    inputs.schema.json      # 入力スキーマ
    inputs.example.json     # 入力サンプル（KIZUNA）
```

## 標準ワークフロー（新規提案を作るとき）

1. `customers/<顧客名>/inputs.json` に顧客データを入れる（電気代・kWh・施設タイプ・稼働・入居率など）。
2. `engine/diagnose.py` で A/Bプランの数値（投資・補助金・自己負担・回収年・20年累計）を算出。
3. `engine/build_proposal.py` で `index.html` を生成（寺嵜流の型に沿ったスライド）。
4. `templates/narration.md` の型でナレーション台本を生成し、`engine/mp3gen.py` でMP3化。
5. 寺嵜がレビュー → 修正点を `knowledge/` に反映（OSが毎回賢くなる）。

## 学習ループ（毎月の更新）

- 成約/失注の理由を `customers/<顧客名>/outcome.md` に記録。
- 効いた表現・数字・補助金情報を `knowledge/` に昇格。
- テンプレと単価・補助金カレンダーを最新化。
- 「決定率を上げた変更」を README の変更履歴に残す。

## やってはいけないこと

- 補助金の締切・要件を推測で断定しない（必ず一次情報で確認、`subsidies.md` に出典付きで記録）。
- 回収年数・削減額を根拠なく盛らない。前提を `pricing.md` に明記し、再現可能にする。
- 顧客の個人情報・連絡先を外部サービスに送らない。
