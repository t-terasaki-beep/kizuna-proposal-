# terasaki-os-code 立ち上げキット

`terasaki-os-code` を「寺嵜流 提案自動生成OS」として育てるための、新セッション用の設計図一式です。
（このフォルダは kizuna-proposal リポジトリに置いた“持ち出し用”の控えです）

## 中身

- `CLAUDE.md` … 新リポジトリのルートに置く「OSの頭脳」。寺嵜流の型・口調・成果物・構成・運用ループを定義。
- `KICKOFF_PROMPT.md` … 新セッションの最初に貼る指示文。
- `README.md` … この説明。

## 進め方（新セッションで本格着手）

1. **claude.ai/code** で「新しいセッション」を作成。
2. リポジトリ選択で **`t-terasaki-beep/terasaki-os-code`** を選ぶ。
3. 最初のメッセージに `KICKOFF_PROMPT.md` の本文を貼る。
4. 参考資料として、kizuna-proposal の以下を渡すと精度が上がる:
   - `index.html`（スライドの型）
   - `mp3gen_kizuna.py`（ナレーション + edge-tts の方式）
5. セッションが `CLAUDE.md` を作り、雛形 → 診断エンジン → 生成 → 音声 の順で組み上げます。

## なぜこの形か（利益とのつながり）

- 利益は「商談 → 契約 → 工事」で生まれる。OS自体が振り込むわけではない。
- OSの仕事は **提案の質と本数を底上げ**すること。これが売上の天井を上げる正しいレバー。
- KIZUNA級（工事 約2,000〜2,500万円規模）の提案を、顧客データを入れるだけで量産できれば、
  寺嵜の商談キャパが増え、月次の成約本数 × 平均粗利 = 月次利益 を押し上げられる。
- 毎月、成約/失注を記録 → 効いた要素を knowledge/ に昇格 → テンプレと補助金情報を更新。
  この学習ループで決定率を上げ続ける。

## 注意

- 補助金の締切・要件は推測で断定しない。一次情報で確認し出典付きで記録する。
- 回収年数・削減額は根拠とセットで。前提を明記し再現可能にする。
- 顧客の個人情報・連絡先を外部サービスに送らない。
